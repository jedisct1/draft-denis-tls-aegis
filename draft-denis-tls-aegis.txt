           AEGIS Ciphersuites for TLS 1.3, DTLS 1.3 and QUIC
                      draft-denis-tls-aegis-latest

Abstract

   This documents proposes new ciphersuites based on the AEGIS family of
   authenticated encryption algorithms for the TLS 1.3 [RFC8446], DTLS
   1.3 [RFC9147] and QUIC [RFC9000] protocols, .

1.  Introduction

   TODO Introduction:

   Why AEGIS for TLS?

   *  Alternative to AES-GCM, targets the same hardware class, but:

   *  Lower memory requirements (no key schedule, no precomputation
      tables)

   *  Higher usage limits (only dependent on the tag size), doesn’t
      require rekeying as frequently as other options

   *  Higher performance

   Integration is simple, same interface as existing TLS 1.3 ciphers.

3.  TLS 1.3

3.1.  Cryptographic Negotiation

   The cryptographic negotiation as specified in [RFC8446],
   Section 4.1.1 remains the same, with the inclusion of the following
   cipher suites:

   +========================+=============+=========+=================+
   | Cipher suite name      | AEAD        |Hash     | Confidentiality |
   |                        | algorithm   |algorithm| level           |
   +========================+=============+=========+=================+
   | TLS_AEGIS_128L_SHA256  | AEGIS-128L  |SHA256   | 128 bits        |
   +------------------------+-------------+---------+-----------------+
   | TLS_AEGIS_128X2_SHA256 | AEGIS-128X2 |SHA256   | 128 bits        |
   +------------------------+-------------+---------+-----------------+
   | TLS_AEGIS_256_SHA512   | AEGIS-256   |SHA512   | 256 bits        |
   +------------------------+-------------+---------+-----------------+
   | TLS_AEGIS_256X2_SHA512 | AEGIS-256X2 |SHA512   | 256 bits        |
   +------------------------+-------------+---------+-----------------+

                    Table 1: AEGIS-based cipher suites

   As defined in [RFC8446], TLS 1.3 cipher suites denote the
   Authenticated Encryption with Associated Data (AEAD) algorithm for
   record protection and the hash algorithm to use with the HMAC-based
   Key Derivation Function (HKDF).

   For TLS 1.3, DTLS 1.3 and QUIC, AEGIS encryption algorithms MUST be
   used with a 128-bit authentication tag.

   Link to [M23] to justify SHA512.

3.2.  TLS Record Payload Protection

   Record payload protection as defined in [RFC8446] is retained for
   these new cipher suites.

4.  DTLS 1.3

   Same cipher suites as TLS 1.3.

4.1.  DTLS 1.3 Record Number Encryption

   In DTLS 1.3, record sequence numbers are encrypted as specified in
   [RFC9147].

   For AEGIS-128L and AEGIS-256, the mask is generated using the AEGIS
   Stream function with:

   *  a 128-bit tag length

   *  sn_key, as defined in [RFC9147], Section 4.2.3

   *  ciphertext[0..16]: the first 16 bytes of the DTLS ciphertext

   *  nonce_len: the AEGIS nonce length

   The 5-byte mask is computed as follows:

   mask = Stream(5, sn_key, ZeroPad(ciphertext[0..16], nonce_len))

5.  QUIC

   Same cipher suites as TLS 1.3.

5.1.  QUIC Header Protection

   In QUIC, parts of the QUIC packet headers are encrypted as specified
   in [RFC9001].

   For AEGIS-128L and AEGIS-256, the mask is generated using the AEGIS
   Encrypt function with:

   *  a 128-bit tag length

   *  hp_key, as defined in [RFC9001], Section 5.4

   *  sample: the 16 bytes QUIC ciphertext sample

   *  nonce_len: the AEGIS nonce length

   The mask is computed as follows:

   mask = Encrypt("", "", hp_key, ZeroPad(sample, nonce_len))
